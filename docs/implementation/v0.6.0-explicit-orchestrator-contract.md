# v0.6.0 Explicit Orchestrator Contract

v0.6.0 establishes shared Orchestrator authority, decision vocabulary, contract references, and runtime role parity.

It does not persist runtime state or execute artifact registry validation. Those belong to later v0.6 milestones.

## Definition

The Orchestrator is the runtime main agent. It may be the Hermes parent agent, Claude Code command context, Codex main agent, OpenCode primary agent, or manual fallback operator.

Python remains the tool layer for workspace setup, source handling, deterministic checks, validation helpers, audit support, and final rendering. Python must not become the standard full brief-generation runtime again.

## Scope

v0.6.0 should make every supported runtime entry point communicate the same control model:

- The current main agent is the Orchestrator.
- Specialist roles are delegated subagents.
- The Orchestrator checks readiness before delegation.
- The Orchestrator validates expected artifacts after delegation.
- The Orchestrator decides whether to continue, retry, request human review, block, or finalize.
- Finalize happens only after audit readiness.

## Contract References

The initial contract sources should be minimal and public-safe:

- Orchestrator authority and allowed responsibilities.
- Stage ordering at the category level.
- Artifact expectations at the category level.
- Default policy-pack shell.

These sources should preserve future provenance needs without implementing provenance. Avoid designs that prevent later representation of:

- artifact identity
- producer stage or role
- consumer stage or role
- validation result summary
- blocking reason
- retry or human-review decision
- decision category attached to a stage

The goal is compatibility with later provenance work, not a provenance graph in v0.6.0.

## Decision Vocabulary

v0.6.0 should standardize the public decision vocabulary used by runtime entries:

- `continue`
- `retry_stage`
- `delegate_repair`
- `request_human_review`
- `block_run`
- `finalize`

These are control decisions for the Orchestrator. They are not yet a persisted event log.

## Runtime Parity

The same Orchestrator authority model should be visible in:

- Hermes parent-agent guidance.
- Claude Code command guidance.
- Codex generated Orchestrator agent config.
- OpenCode primary Orchestrator agent config.
- Manual fallback handoff text.
- Generated agent documentation.

Runtime-specific mechanics can differ, but business artifact expectations should not fork by runtime.

## Non-Goals

- No full DAG runtime.
- No workflow-state persistence beyond contract references in handoff artifacts.
- No artifact registry validator execution.
- No evidence or execution provenance graph.
- No feedback issue or repair-plan loop.
- No public golden-case set.
- No rewrite of final report rendering.
- No expansion of search providers.

## Public Acceptance

v0.6.0 is ready when:

- Runtime entries identify the Orchestrator as main agent.
- Generated platform adapter files are consistent with the canonical role source.
- Handoff artifacts point the runtime main agent to contract references.
- The standard `run` command still creates handoff artifacts and does not generate a brief.
- Regression coverage prevents restoring the old Python full-pipeline path.
- Public docs describe high-level contract behavior without publishing private implementation detail.
