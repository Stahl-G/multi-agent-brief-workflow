# MABW v0.8.5 Preview — 080 Experiment Harness

MABW v0.8.5 is a preview release for MABW-080 experiment harness tooling.

It makes experiment runs easier to register and compare under the same frozen
fact layer, but it does not claim that output quality improved, that guidance
caused better prose, that sources semantically prove claims, or that model
performance has been measured.

## What's New

- **MABW-080 scorecard drafts.** `multi-agent-brief experiments 080 score-run`
  builds deterministic scorecard metadata from a registered run, case
  definition, archive/control projections, timing, reader-clean status, gate
  status, and fact-layer comparison.

- **MABW-080 assessment import.**
  `multi-agent-brief experiments 080 import-assessment` imports externally
  supplied guidance-manifestation assessment into a scorecard. Python validates
  identity binding, guidance-entry coverage, score vocabulary, and assessment
  method, then derives A/B/invalid validity classes from deterministic control
  fields plus the imported metadata.

- **MABW-080 case summaries.** `multi-agent-brief experiments 080 summarize`
  aggregates existing scorecards into condition counts, A/B/invalid counts,
  manifestation-score counts, reader-clean rates, coverage-delta status, timing
  status, and invalid reasons.

- **MABW-080 condition scaffolds.**
  `multi-agent-brief experiments 080 scaffold-condition` prepares a condition
  workspace by importing the frozen fact layer through the deterministic
  fast-rerun transaction and writing operator instructions. It does not run
  subagents, gates, finalize, registration, scoring, or summarization.

- **Delivery snapshot convenience copies.** `multi-agent-brief finalize` still
  refreshes `output/delivery/` as the latest reader surface, and now also writes
  reader-facing copies under `output/delivery-history/<run_id-or-timestamp>/`.

## MABW-080 Preview Boundary

The 080 harness is experiment metadata infrastructure:

- `validate-case` checks whether a case can support a manifestation experiment.
- `register-run` registers a completed workspace run as experiment metadata.
- `score-run` builds deterministic control/readiness fields.
- `import-assessment` merges externally supplied assessment scores.
- `summarize` aggregates existing scorecards. Invalid runs are failure evidence
  and are excluded from interpretable A/B metric denominators.
- `scaffold-condition` imports the same frozen fact layer into a condition
  workspace and writes operator instructions. It does not execute the run.

Python does not judge guidance manifestation, prose quality, taste, factual
regression, reader preference, or semantic source support.

An `A_controlled` scorecard still depends on a clean control trace, matching
frozen fact layer, passing reader/gate/finalize/archive projections, available
timing, and human or LLM-assisted human review for relevant guidance scores.
`llm_only` assessment can support integration-level metadata, but not
`A_controlled`.

## Delivery Snapshot Boundary

`output/delivery-history/` is a user convenience surface. It helps users keep
visible weekly/report outputs from overwriting each other during repeated
`finalize` runs.

It is not the immutable control archive. The authoritative archive remains:

```bash
multi-agent-brief state finalize-complete --workspace <workspace>
```

and is written under:

```text
output/runs/<run_id>/
```

## Evidence

- Full deterministic test suite runs in CI without LLM calls.
- CI includes generated-agent parity, runtime asset parity, version consistency,
  release consistency, public-safety checks, source-provider coverage, and
  package-install smoke tests.
- Regression coverage includes scorecard archive identity/hash validation,
  assessment identity binding, guidance-entry coverage, A/B/invalid validity
  class derivation, output idempotency, and delivery snapshot overwrite safety.

## What This Preview Does Not Claim

MABW v0.8.5 does not claim output-quality improvement, semantic support
verification, model-performance measurement, faster end-to-end runtime, Codex
parity, or autonomous repair.

## Get Started

Install from source:

```bash
git clone https://github.com/Stahl-G/multi-agent-brief-workflow.git
cd multi-agent-brief-workflow
bash scripts/setup.sh
```

Create a real brief workspace:

```bash
multi-agent-brief onboard
multi-agent-brief init ~/mabw-workspace --from-onboarding onboarding.json
multi-agent-brief run --workspace ~/mabw-workspace
```

080 experiment metadata flow:

```bash
multi-agent-brief experiments 080 validate-case <case_dir>
multi-agent-brief experiments 080 scaffold-condition \
  --case <case_dir> \
  --condition memory \
  --workspace <condition_workspace> \
  --archive <source_run_archive_manifest>
multi-agent-brief experiments 080 register-run \
  --case <case_dir> \
  --condition memory \
  --workspace <workspace> \
  --output run_record.json
multi-agent-brief experiments 080 score-run \
  --case <case_dir> \
  --run-record run_record.json \
  --output scorecard.json
multi-agent-brief experiments 080 import-assessment \
  --scorecard scorecard.json \
  --assessment assessment.json \
  --output assessed_scorecard.json
multi-agent-brief experiments 080 summarize \
  --case <case_dir> \
  --output case_summary.json
```
