# How I Use BriefLoop Each Week

This is not a trust map and not a control-surface reference. It is a habit
script: how to use BriefLoop in a weekly briefing rhythm.

## Seven-Sentence Version

Monday: run `/briefloop run`, let the workflow turn the week's materials into an
auditable draft, and go to meetings.

Tuesday: read the draft and send awkward parts to `/briefloop feedback` in plain
language. It takes two minutes.

Wednesday: review the feedback split: approve what should be remembered, route
what needs checking, and turn repeatable format needs into templates instead of
memory.

Thursday: rerun once and specifically check whether last week's approved
guidance entered the new snapshot. If unsure, use `/briefloop status`.

Friday: run `/briefloop deliver`. If gates are green, deliver only
`output/delivery/brief.md` and `output/delivery/<named-brief>.docx`. The source
appendix is appended to the delivery file; standalone `output/source_appendix.md`
is an audit/control copy, not a separate reader delivery.

Saturday: leave it alone.

Sunday: leave it alone. It keeps the stage, sources, preferences, and gate
records visible so you can actually rest.

## Four Things To Remember

| What you care about | Where to look |
|---|---|
| Where this run is | `/briefloop status <workspace>` |
| Where each number came from | Claim Ledger, source appendix, audit and gate results |
| What it learned | Improvement Ledger; only approved preferences take effect |
| What is guarding delivery | Gate, reader-clean, and finalize-complete results |

## Two Habits

First, do not call fact problems preferences. If a number is wrong, a source is
missing, or a date is stale, route it through this-run checking and repair.

Second, do not store fixed formatting as long-term memory. Structure that must
appear every time should become a template, rule, or delivery standard.
